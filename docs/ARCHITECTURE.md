# 工程架构

## 1. 目标

第一期呈现《琵琶行》正文全篇。作品内容、舞台编排、物理、渲染、镜头与音频保持分层，避免再次把所有逻辑堆进单一舞台类。

后续应能低成本支持：

- 新增《长恨歌》《将进酒》等作品；
- 同一作品切换不同视觉主题；
- 琵琶、古琴、编钟等不同声音配置；
- 文字弦幕、横向水波、卷轴等不同呈现模式；
- 桌面端和移动端采用不同网格密度；
- 增加导览、逐段阅读、自动演奏和作品选择。

第一期已经确定采用“连续横向长卷 + 巨型文字帘”：正文组成一整片连续物理空间，每列容纳两段原文，采用古籍竖排并从右向左阅读。

## 2. 分层

```text
app
 ├── features
 │    └── poem-stage
 ├── content
 ├── themes
 └── core
      ├── physics
      ├── rendering
      ├── viewport
      └── audio
```

### app

负责创建页面 DOM、获取必要元素并启动功能模块。这里不写物理、渲染或交互算法。

### content

只保存作品数据，例如作者、朝代、序文、正文段落、朗读顺序和内容版本。不得依赖 Canvas、DOM 或声音模块。

### themes

当前保存《琵琶行》的旋律短句、五声音阶和列到音高的映射。视觉令牌在 `styles/tokens.css`，舞台尺寸由功能层的 `stageMetrics.ts` 计算。

### core

可脱离《琵琶行》独立复用的底层引擎：

- `physics`：向量、粒子、距离约束和模拟器；
- `rendering`：字形缓存、Canvas 尺寸与绘制；
- `viewport`：世界坐标、镜头位置、横向长卷和可见区域；
- `audio`：音频上下文、音色、复音限制和触发节流。

### features

把内容、主题和底层能力组合成用户可见功能。`poem-stage` 内部进一步分为：

- `PoemStage`：生命周期、拨弦与声音编排；
- `StageNavigation`：边缘巡航、时间轴、键盘、自动播放和曲终门槛；
- `StageUiPresenter`：卷首、章节、进度和曲终 DOM 状态；
- `poemColumns`：正文分列与完整性统计；
- `stageMetrics`：桌面和移动端的响应式舞台尺寸。

## 3. 依赖规则

允许：

```text
app -> features
features -> content/themes/core
themes -> content types only when needed
core -> core
```

禁止：

- `core` 引用具体作品；
- `content` 操作 DOM、Canvas 或 AudioContext；
- `themes` 包含页面业务流程或 DOM 操作；
- `physics` 直接触发声音；
- `rendering` 自己读取鼠标事件；
- `viewport` 修改作品原文或物理参数；
- 在一个巨型 `main.ts` 中堆叠所有逻辑。

物理引擎只产生位置和碰撞事件，由 `poem-stage` 决定是否播放声音或更新界面。

## 4. 运行链路

```text
WorkDefinition + melody mapping
        ↓
PoemStage
        ├── StageNavigation → ScrollViewport
        ├── StringSimulation
        ├── StageRenderer
        └── StageUiPresenter
        ↓
Pointer / Keyboard → PipaEngine
```

## 5. 长卷与巨型文字帘

### 空间模型

- 正文按古籍竖排拆成连续文字列，每列默认合并两段原文；
- 第一列位于世界坐标最右侧，阅读方向为从右向左；
- 屏幕只是长卷世界中的一个观察窗口；
- 抵住左右卷缘、拖动底部卷轴或使用方向键改变镜头位置，不重新排列原文；
- 不设置按钮式开始页；开场题字与正文处于同一个连续空间，第一次有效左滑直接驱动镜头并让题字渐隐退场；
- 空白拖动和滚轮不移动画卷，避免与页面常规滚动习惯冲突；
- 每一列既是阅读单位，也是可拨动的物理弦列。

### 性能策略

《琵琶行》全文不能让所有字符在所有时刻都进行最高频率物理计算，因此采用分级运行：

1. 屏幕内文字：完整物理、绘制和声音响应；
2. 屏幕两侧缓冲区：继续物理模拟，保证进入画面时运动连续；
3. 远离屏幕的文字：保留静态状态或低频更新；
4. 再次进入缓冲区时恢复完整模拟；
5. 用户快速横向移动时优先保证滚动和绘制流畅度。

这属于视口虚拟化，不会删减或分页原文，用户感知到的仍是一整面连续文字帘。

## 6. 扩展方式

### 新增一首诗

1. 在 `src/content/works/` 增加作品文件；
2. 在 `src/themes/<work-id>/` 增加旋律映射；
3. 在 `public/assets/` 放置视觉或授权音频资产；
4. 由 `app/bootstrap.ts` 选择并装配作品；
5. 不修改物理引擎。

### 新增一种乐器

1. 在 `core/audio` 增加独立引擎或将现有 `PipaEngine` 抽象为共享合成层；
2. 配置音阶、泛音、衰减和噪声特征；
3. 必要时增加有明确许可的采样资源；
4. 不修改物理和渲染模块。

### 新增一种物理表现

实现统一的模拟接口，由功能层选择 `cloth`、`strings` 或其他模拟器，不改变作品数据。

## 7. 性能原则

- 物理循环与 DOM 更新分离；
- Canvas 使用 `requestAnimationFrame`；
- 字形预渲染并缓存，不在每帧重复 `fillText`；
- 根据设备性能调整粒子数量和 DPR；
- 音频限制最大同时发声数并设置触发冷却；
- 只对视口及缓冲区内的文字进行完整物理计算；
- 支持 `prefers-reduced-motion`，关闭环境漂移、涟漪和装饰性位移；
- 所有主要阅卷操作提供鼠标、触摸和键盘路径。
